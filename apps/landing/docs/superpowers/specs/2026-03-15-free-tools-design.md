**Date:** 2026-03-15
**Status:** Draft
**Scope:** 5 interactive client-side tools + Tools Hub + ~270 programmatic SEO pages

## 1. Overview

Build 5 free developer tools as an SEO & lead generation engine for better-i18n.com. All tools are client-side only, ungated (no email capture), and follow the existing landing site's design system and architecture patterns.

### Tools

| # | Tool | Path | Est. Monthly Search | Programmatic Pages |
|---|------|------|--------------------|--------------------|
| 1 | Locale Explorer | `/tools/locale-explorer` | 5K-15K | ~250 (per locale) |
| 2 | ICU Message Playground | `/tools/icu-playground` | 500-2K | 0 |
| 3 | Translation File Converter | `/tools/translation-file-converter` | 3K-8K | 16 (per format pair) |
| 4 | Localization Cost Calculator | `/tools/cost-calculator` | 2K-5K | 0 |
| 5 | Hreflang Tag Generator | `/tools/hreflang-generator` | 5K-10K | 0 |

**Total indexable pages:** ~270+ (5 tool pages + 1 hub + ~250 locale pages + 16 converter pages)

### Goals

- **SEO:** 13K-33K monthly organic search potential
- **Funnel coverage:** Top (Locale Explorer) → Mid (ICU, Converter) → Bottom (Cost Calculator)
- **AI visibility:** All tools registered in llms.txt
- **Brand:** Position Better i18n as the developer-first i18n platform

## 3. Tool Specifications

### 3.1 Locale Explorer

**Main page (`/tools/locale-explorer`):**
- Searchable/filterable table of ~250 locales
- Search: language name, locale code, country
- Filters: RTL/LTR, script type, region
- Columns: locale code, language, country, script, direction, sample date/number format
- Click row → detail page

**Detail page (`/tools/locale-explorer/$localeCode`):**
- Locale metadata: language, region, script, writing direction
- Live Intl API output:
  - `Intl.DateTimeFormat` (short, medium, long, full)
  - `Intl.NumberFormat` (decimal, currency, percent, unit)
  - `Intl.RelativeTimeFormat` (past/future examples)
  - `Intl.ListFormat` (conjunction, disjunction)
  - `Intl.DisplayNames` (language, region, script names)
- CLDR plural rules: categories (one, few, many, other) with numeric examples
- Approximate speaker population (hardcoded in `locales.ts` from Ethnologue/Wikipedia data — static, updated manually per release, not fetched from API)
- Framework config snippets (next-intl, react-intl, i18next, vue-i18n)
- Related locales table (same language, different regions)

**Programmatic pages:** ~250 locales from CLDR data, defined as a static array in `src/lib/tools/locales.ts` at build time. This file is the single source of truth for all locale codes used by the explorer detail pages and the locale selector components.

**No thin content risk:** Each page has 6+ Intl API format examples, plural rules, framework snippets, and related locales — all unique per locale.

### 3.2 ICU Message Playground

- **Left panel:** ICU message editor with syntax highlighting (CodeMirror)
- **Right panel:** Live preview output
- **Variable injection:**
  - `{count}` → numeric slider
  - `{gender}` → dropdown (male/female/other)
  - `{date}` → date picker
  - Custom variables → text input
- **Multi-locale preview:** Same message rendered in 3-4 locales simultaneously (shows plural rule differences)
- **Error explanations:** Not just "syntax error" but descriptive messages like "Missing required `other` clause in plural statement"
- **Shareable URL:** ICU message + variable values encoded in URL hash (no server needed)
- **Code snippets:** Copy-paste ready code for react-intl, next-intl, i18next, vue-i18n
- **Examples library:** Pre-loaded examples (plurals, select, selectordinal, nested, date/number formatting)

### 3.3 Translation File Converter

**Hub page (`/tools/translation-file-converter`):**
- Format matrix grid showing JSON as the hub format — JSON row and JSON column are active, all other cells are greyed out with tooltip "Coming soon" (only JSON↔X pairs are supported)
- Active cells link to their format pair page (e.g., JSON→PO links to `/tools/translation-file-converter/json-to-po`)
- Supported formats: JSON, PO/POT, XLIFF, ARB, YAML, CSV, Android XML, iOS .strings, Java .properties
- Brief description of each format with use cases

**Format pair page (`/tools/translation-file-converter/$pair`):**
- **Left panel:** Input (file upload drag-and-drop OR paste text)
- **Right panel:** Converted output (preview + download button)
- **Format-specific options:**
  - JSON: flat vs nested, indent (2/4/tab)
  - PO: include comments, msgctxt handling
  - XLIFF: version 1.2 vs 2.0
  - ARB: include @metadata
  - YAML: flow vs block style
  - CSV: delimiter, quote style
  - Android XML: translatable attribute
  - iOS .strings: encoding
  - Properties: unicode escape
- **Format pair SEO content:** "When to convert JSON to PO", use cases, format comparison

**16 format pairs (9 formats total — JSON as hub, 8 other formats × 2 directions = 16 pairs):**
- json-to-po, po-to-json
- json-to-xliff, xliff-to-json
- json-to-arb, arb-to-json
- json-to-yaml, yaml-to-json
- json-to-csv, csv-to-json
- json-to-android-xml, android-xml-to-json
- json-to-ios-strings, ios-strings-to-json
- json-to-properties, properties-to-json

### 3.4 Localization Cost Calculator

**Step-based UI:**

**Step 1 — Project Info:**
- Word count: manual input OR file upload (parse JSON/PO/XLIFF to count words)
- Source language selection
- Number of translation keys (auto-calculated from file)

**Step 2 — Target Languages:**
- Multi-select from locale list (reuses Locale Explorer data)
- Quick presets: "European (10 languages)", "APAC (8)", "Global Top 20"
- Shows selected count and total word volume

**Step 3 — Results Dashboard:**
- 3-column cost comparison:
  - Professional Human Translation ($0.10-0.25/word)
  - Human + AI Review ($0.05-0.12/word)
  - Better i18n AI ($0.01-0.03/word)
- Per-language cost breakdown table
- Monthly maintenance cost projection (assumes 10-20% content change/month)
- ROI projection chart: revenue uplift from new markets vs translation cost
- Break-even timeline visualization
- Total savings highlight: "Save X% with Better i18n vs traditional translation"

**Pricing data:** Static industry benchmarks, hardcoded in `cost-data.ts`. Better i18n pricing from config.

### 3.5 Hreflang Tag Generator

- **URL input:** Single URL or bulk CSV upload
- **Locale selection:** Multi-select with search (reuses locale data)
- **x-default handling:** Auto-selects primary locale, configurable
- **Output formats:**
  - HTML `<link rel="alternate">` tags
  - XML sitemap `<xhtml:link>` entries
  - HTTP header format
- **Validation:**
  - Missing self-referencing tag warning
  - Invalid locale code detection
  - Duplicate hreflang detection
  - Missing x-default warning
- **Bulk mode:** Upload CSV (URL, locale pairs), generate complete sitemap fragment
- **Copy button + download:** One-click copy or download as file

### Breadcrumb Integration (page-seo.ts)

Add `"tools"` to `CATEGORY_LABELS`:
```typescript
const CATEGORY_LABELS = {
  "i18n": { label: "i18n Guides", href: "/i18n" },
  "compare": { label: "Comparisons", href: "/compare" },
  "blog": { label: "Blog", href: "/blog" },
  "features": { label: "Features", href: "/features" },
  "tools": { label: "Free Tools", href: "/tools" },  // NEW
};
```

**3-level breadcrumb support:** The current `getBreadcrumbItems()` only handles 1 or 2 segments (it takes `segments[0]` as category and `segments[segments.length - 1]` as page, silently dropping middle segments). Tool detail pages like `tools/locale-explorer/ja-JP` have 3 segments. Extend the function to handle arbitrary depth:

```typescript
// For paths with 3+ segments: Home > Category > Subcategory > ... > Page
// Example: tools/locale-explorer/ja-JP → Home > Free Tools > Locale Explorer > ja-JP
if (segments.length >= 3) {
  // First segment: category (from CATEGORY_LABELS)
  items.push({ label: categoryLabel, href: category?.href ?? `/${segments[0]}` });
  // Middle segments: subcategories with links
  for (let i = 1; i < segments.length - 1; i++) {
    const subPath = segments.slice(0, i + 1).join("/");
    const subLabel = messages[`breadcrumbs.${segments[i]}`]
      ?? segments[i].replace(/-/g, " ").replace(/\b\w/g, (c) => c.toUpperCase());
    items.push({ label: subLabel, href: `/${subPath}` });
  }
  // Last segment: current page (no href)
  const pageLabel = messages[`breadcrumbs.${segments[segments.length - 1]}`]
    ?? segments[segments.length - 1].replace(/-/g, " ").replace(/\b\w/g, (c) => c.toUpperCase());
  items.push({ label: pageLabel });
}
```

### Build-Time Page Generation (generate-pages.ts)

Add two new generator functions following the existing `generateBlogDetailPages()` pattern.

**Integration point:** Called in `generatePages()` (line 594), not `generateAllPages` (which doesn't exist):
```typescript
export function generatePages(data: SeoData, buildDate?: string): readonly PageEntry[] {
  const { locales, blogPosts, featurePages, localeCoverage } = data;
  const marketingPages = generateMarketingPages(locales, buildDate, localeCoverage);
  const blogPages = generateBlogPages(blogPosts, locales);
  const featureDetailPages = generateFeatureDetailPages(featurePages, locales);
  const localeExplorerPages = generateLocaleExplorerPages(locales);     // NEW
  const converterPairPages = generateConverterPairPages(locales);       // NEW
  const allPages = [...marketingPages, ...blogPages, ...featureDetailPages,
                     ...localeExplorerPages, ...converterPairPages];
  // ...
}
```

No changes to `SeoData` interface needed — both generators use hardcoded data (locale codes from `locales.ts`, format pairs from `formats.ts`), not API-fetched data.

**`generateLocaleExplorerPages(allLocales)`:**
- Imports locale code list from `src/lib/tools/locales.ts` (hardcoded static array)
- For each locale code (ja-JP, en-US, etc.) × each site locale:
  - Tier 1-2 site locales → sitemap entry with priority 0.6, changefreq yearly, prerender enabled
  - Tier 3 site locales → explicit `PageEntry` with `noindex: true`, `priority: 0`, no prerender (matches existing blog tier 3 pattern)
- Generates hreflang alternateRefs across non-tier3 site locales

**`generateConverterPairPages(allLocales)`:**
- Imports format pair list from `src/lib/tools/formats.ts` (hardcoded: `["json-to-po", "po-to-json", ...]`)
- For each pair × each site locale:
  - Tier 1-2 site locales → sitemap entry with priority 0.7, changefreq monthly, prerender enabled
  - Tier 3 site locales → explicit `PageEntry` with `noindex: true`, `priority: 0`, no prerender (same tier 3 pattern)

### Lazy Loading & Progressive Enhancement

All heavy libraries use dynamic `import()` wrapped in `React.lazy()` + `Suspense`:

**Suspense fallback:** A skeleton UI matching the tool's layout — shows heading, description, FAQ, and a centered loading spinner where the interactive tool will appear. This ensures:
1. SSR renders full SEO content (title, description, FAQ, structured data) without JS
2. Hydration shows a meaningful loading state while libraries download
3. If library fails to load, an error boundary shows "Tool failed to load. Please refresh." with a retry button

## 8. Testing Strategy

### Unit Tests
- ICU parser: valid/invalid message parsing, error messages, variable extraction
- File converters: each format pair round-trip (parse → serialize → parse = same data)
- Hreflang generator: tag generation, validation rules, edge cases
- Cost calculator: pricing calculations, word counting from files
- Locale data: data integrity, all codes valid BCP 47

### Integration Tests
- Each tool page renders without errors
- Route parameters resolve correctly (locale codes, format pairs)
- SEO head data generated correctly (meta, canonical, hreflang, structured data)
- Breadcrumbs render correctly at each level

### E2E Tests (Critical paths)
- Locale Explorer: search → filter → click detail → verify Intl output
- ICU Playground: type message → inject variables → see output → share URL → reload → same state
- File Converter: upload file → select format → convert → download
- Cost Calculator: enter words → select languages → see comparison
- Hreflang Generator: enter URL → select locales → copy output

## 9. Performance Considerations

- **Prerender:** All tool pages (Tier 1-2) built as static HTML at build time
- **Lazy loading:** All heavy libraries (CodeMirror, parsers, recharts) loaded only on relevant tool pages
- **Code splitting:** Each tool is a separate route → automatic code split by TanStack Router
- **CLDR data:** Locale database (~250 entries) loaded once, shared across Locale Explorer and Cost Calculator locale selectors
- **No API calls:** Everything client-side, zero server cost, instant response
- **Progressive enhancement:** Tool pages render SEO content (description, FAQ, how-to) without JavaScript, tools activate when JS loads
